    # Drift Operations Runbook

    ## 1. Overview
    - What: Operational guide for running the Drift research agent in production
    - Who: Stefano (sole operator)
    - Companion: `docs/setup.md` covers initial project setup; this runbook covers day-to-day operations

    ## 2. GitHub Actions Setup (Day 1)
    Per D-11, this is included in the runbook, not delegated to setup.md.
    - Navigate to repository Settings > Secrets and variables > Actions
    - Add these four repository secrets:
      - `ANTHROPIC_API_KEY` — from Anthropic Console > API Keys
      - `SUPABASE_URL` — from Supabase Dashboard > Settings > API > Project URL
      - `SUPABASE_SERVICE_KEY` — from Supabase Dashboard > Settings > API > service_role key (NOT anon key)
      - `FIRECRAWL_API_KEY` — from Firecrawl Dashboard > API Keys (optional — agent works without it)
    - Verify workflow file exists at `.github/workflows/agent-run.yml`
    - Workflow schedule: Runs automatically on 1st and 15th of each month at 06:00 UTC
    - Enable GitHub email notifications: Settings > Notifications > Actions > check "Send notifications for failed workflows only"

    ## 3. Manual Agent Trigger
    - Go to Actions tab > "Drift Agent — Bi-weekly Research Run" > "Run workflow"
    - Optional: Enter a specific company_id to run for one company only
    - Leave company_id blank to run all tracked companies
    - CLI alternative (if you have local env): `python backend/agent.py` or `python backend/agent.py --company-id <id>`

    ## 4. Signal Review Workflow
    Two complementary methods (per D-05):

    ### Admin UI
    - Navigate to `/admin`
    - Log in with Supabase Auth email/password
    - Review each draft signal: read excerpt, classification, agent reasoning
    - Click "Approve" to publish or "Reject" to delete
    - Approved signals appear on public company pages immediately

    ### CLI
    ```bash
    python backend/agent.py --review                    # List all pending signals
    python backend/agent.py --approve <signal-id>       # Publish a signal
    python backend/agent.py --reject <signal-id>        # Delete a draft signal
    ```

    ## 5. Firecrawl Troubleshooting
    Per D-10, cover rate limits, paywalled pages, and fallback behavior.

    ### Rate Limits (Free Tier)
    - Firecrawl free tier: ~500 credits/month
    - Each scrape = 1 credit
    - If rate limited: agent retries with exponential backoff (tenacity library, 3 retries)
    - If all retries fail: agent falls back to Claude web search (no Firecrawl)
    - Monitor: Check agent_runs table for `run_summary` mentioning Firecrawl failures

    ### Paywalled Pages
    - Some IR pages require login (Bloomberg, Refinitiv, etc.)
    - Firecrawl cannot access paywalled content
    - Agent automatically falls back to web search for paywalled pages
    - Documented limitation: coverage may be lower for companies with paywalled IR pages

    ### Firecrawl Down / API Errors
    - If FIRECRAWL_API_KEY is not set: agent works normally without Firecrawl
    - If Firecrawl API returns errors: agent continues with web search fallback
    - No manual intervention needed — agent is designed to degrade gracefully

    ## 6. Agent Failure Recovery
    - Check GitHub Actions run log for error details
    - Common failures:
      - `ANTHROPIC_API_KEY invalid` — Regenerate key in Anthropic Console, update GitHub secret
      - `SUPABASE_SERVICE_KEY invalid` — Regenerate in Supabase Dashboard, update GitHub secret
      - `Timeout after 30 minutes` — Agent may be processing too many companies; run individually with `--company-id`
      - `Rate limit exceeded` — Wait for rate limit reset (usually 1 hour), then re-trigger manually
    - Agent writes status to `agent_runs` table even on failure — check via admin UI or:
      ```sql
      SELECT id, status, run_summary, created_at
      FROM agent_runs
      ORDER BY created_at DESC
      LIMIT 5;
      ```

    ## 7. Log Inspection
    - GitHub Actions: Actions tab > click workflow run > click "research-run" job > expand step logs
    - Local runs: stdout/stderr (redirect with `python backend/agent.py 2>&1 | tee agent.log`)
    - Database: `agent_runs` table stores run_summary, signals_proposed, signals_approved, estimated_cost_usd

    ## 8. Supabase Query Basics
    Useful queries for operational monitoring:
    ```sql
    -- Count draft signals pending review
    SELECT COUNT(*) FROM signals WHERE is_draft = true;

    -- Recent agent runs with status
    SELECT id, status, signals_proposed, signals_approved, estimated_cost_usd, created_at
    FROM agent_runs ORDER BY created_at DESC LIMIT 10;

    -- Check company last research run
    SELECT name, ticker, last_research_run FROM companies WHERE tracking_active = true;

    -- Signals by classification for a company
    SELECT classification, COUNT(*) FROM signals
    WHERE company_id = '<id>' AND is_draft = false
    GROUP BY classification ORDER BY count DESC;
    ```

    ## 9. Admin Authentication Setup
    - Create admin user in Supabase Dashboard: Authentication > Users > "Add User"
    - Enter email and password
    - User can now log in at `/admin`
    - Only one admin user needed (Stefano)

    ## 10. Monetization Gate — Scaling Criteria

    Per D-16, D-17, D-18: The gate to adding company #2 is public launch readiness, NOT a revenue milestone.

    ### Conditions (ALL must be met):
    1. **Editorial maturity** — The Drift site is mature enough to show publicly. Sandoz company page has no rough edges, all objectives have evidence, graveyard entries are classified correctly.
    2. **Ad slot readiness** — Ad slot placeholders are in place on landing page (slots 1-2) and company page (slots 3-4). They look professional and intentional.
    3. **Agent stability** — At least 2 consecutive clean bi-weekly agent runs on Sandoz with no critical errors. Check `agent_runs` table: `SELECT status FROM agent_runs ORDER BY created_at DESC LIMIT 2;` — both should be "completed".
    4. **Runbook reviewed** — This runbook has been read and understood by Stefano.

    ### Decision process:
    - This is a qualitative judgment call made by Stefano (per D-18)
    - No automated trigger — Stefano decides when to add company #2
    - When ready: `python backend/agent.py --intake <new-company-id>` to onboard

    ### What happens after the gate:
    - Add 1-2 companies (Roche, Volkswagen, or BP suggested in CLAUDE.md)
    - Monitor agent runs for 2 weeks
    - Begin ad network integration (Carbon Ads / EthicalAds) for revenue
    - Consider premium subscription tier (Stripe) at ~100 visitors/month
