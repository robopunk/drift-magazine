## Plan 04-03: Production Authentication Verification

**Wave:** 2 (depends on 04-01 for frontend env vars)
**Autonomous:** Yes (with checkpoint for user email entry)
**Duration:** ~20 minutes

### Objective

Create an admin user in Supabase Auth, test the login/logout flow on the locally-running frontend, and verify that unauthenticated users cannot access `/admin`.

This plan satisfies **AUTH-01** (Supabase Auth gate on `/admin` works), **AUTH-02** (admin user can log in with email/password), and **AUTH-03** (unauthenticated users see 403/redirect on `/admin`).

### Context

**What exists:**
- `frontend/src/app/admin/page.tsx` — admin page, auth-gated via Supabase Auth
- `frontend/src/lib/auth.ts` — authentication utilities (session management, login/logout)
- `frontend/.env.local` — Supabase credentials (from Plan 04-01)
- Supabase Dashboard → Authentication → Users — where we create the admin account

**What's needed:**
- Create an admin user account in Supabase (email/password)
- Start the frontend dev server
- Test login with the new admin credentials
- Test logout
- Test that `/admin` redirects unauthenticated users

### Tasks

#### Task 1: Create admin user in Supabase Dashboard

**Files:** (Supabase UI only, no file changes)

**Action:**
1. Go to [Supabase Dashboard](https://supabase.com/dashboard)
2. Select your Drift project
3. Navigate to **Authentication** → **Users**
4. Click **+ Create new user**
5. Fill in:
   - **Email:** Enter your email (e.g., `stefano@example.com`)
   - **Password:** Create a strong password (min 8 chars, recommend 12+)
   - **Auto Confirm user:** Check this box (so the account is immediately active)
6. Click **Create user**

The user will appear in the Users list. You'll use this email/password to test login.

**Verify:**
The user appears in the Supabase Users list with:
- Email filled in
- Created date/time visible
- Status: "Confirmed" (because we auto-confirmed)

**Done:** Admin user created in Supabase Auth with email and password set.

#### Task 3: Test unauthenticated access to /admin (should redirect/block)

**Files:** (read-only)
- `frontend/src/app/admin/page.tsx`
- `frontend/src/lib/auth.ts`

**Action:**
With the dev server running, open your browser and navigate to http://localhost:3000/admin **without logging in first**.

Expected behavior: You should be redirected to a login page (Supabase Auth UI) or see a 403 error, depending on the implementation.

Expected redirects:
- If using Supabase's built-in auth UI: redirects to `/auth/login` or Supabase's hosted UI
- If using a custom login form: shows login form on `/admin` or redirects to a login page
- If using a middleware check: returns 403 Forbidden

The key is: **you cannot view the admin page without being authenticated**.

**Verify:**
```bash
# Test unauthenticated access
curl -s http://localhost:3000/admin | grep -iE "(login|authenticate|forbidden|403)" | head -1
# Should find at least one of these keywords

# Or check the response code
curl -s -o /dev/null -w "%{http_code}" http://localhost:3000/admin
# Should output 301 (redirect) or 403 (forbidden) — NOT 200 (OK)
```

**Done:** `/admin` is not accessible without authentication (returns 3xx/403, not 200).

#### Task 4: Log in with the admin user and verify access

**Files:** (read-only)
- `frontend/src/app/admin/page.tsx`
- `frontend/src/lib/auth.ts`

**Action:**
In your browser, you should still be on the login page (or redirected from `/admin`). Log in using the credentials from Task 1:
- Email: (the email you created in Supabase)
- Password: (the password you set)

Click "Sign In" or equivalent button. After successful login, you should be redirected to the `/admin` page and see the admin interface (whatever controls/data are displayed there).

Expected behavior:
1. Login form accepts email/password
2. After submission, the page processes the login
3. Redirects to `/admin`
4. Admin page loads and displays content (user name, logout button, company management UI, etc.)
5. The browser's Storage/Cookies should now have a session token

**Verify:**
In your browser's **Developer Tools** → **Application** → **Cookies** (or **Storage**):
- Look for cookies from `localhost` related to Supabase or authentication
- Should see something like `sb-...` (Supabase session cookies) or similar
- In browser console: `sessionStorage` or `localStorage` should have Supabase session data

**Done:** Admin user successfully logged in, `/admin` page is now accessible and displays admin interface.

#### Task 5: Test logout and re-verify /admin is blocked

**Files:** (read-only)
- `frontend/src/app/admin/page.tsx`
- `frontend/src/lib/auth.ts`

**Action:**
On the `/admin` page, find and click the **Logout** button (usually in the header or footer). After clicking, you should be logged out and redirected back to the landing page or login page.

Then, navigate back to http://localhost:3000/admin. You should again be redirected or blocked (same behavior as Task 3 — unauthenticated access is denied).

**Verify:**
After logout:
1. Check browser storage (Developer Tools → Application) — Supabase session cookies should be cleared
2. Navigate to http://localhost:3000/admin — should redirect/block again

```bash
# After logging out and clearing cookies, test unauthenticated access again
curl -b "" -s -o /dev/null -w "%{http_code}" http://localhost:3000/admin
# Should return 301 or 403, not 200
```

**Done:** Logout works, session is cleared, `/admin` is again inaccessible without authentication.

```markdown
### Admin Authentication

The `/admin` page is protected by Supabase Auth (email/password).

**To create an admin user:**
1. Go to Supabase Dashboard → Authentication → Users
2. Click "Create new user"
3. Enter email and password
4. Check "Auto Confirm user"
5. Click "Create user"

**To log in (development):**
1. Navigate to http://localhost:3000/admin (or production URL)
2. You'll be redirected to the login form
3. Enter email and password
4. Click "Sign In"

**To log in (production):**
1. Navigate to https://yourdomain.com/admin
2. Same login flow as development

**To log out:**
1. Click the "Logout" button on the admin page
2. You'll be redirected to the landing page
3. Session will be cleared

**Security notes:**
- Passwords are stored encrypted in Supabase Auth
- Sessions are stored in browser localStorage (encrypted)
- Unauthenticated users cannot access `/admin` (redirected to login)
- Admin page is only for trusted operators (Stefano, your team)

**To reset an admin password:**
1. Go to Supabase Dashboard → Authentication → Users
2. Find the user
3. Click the user row
4. Click "Reset password" — user will receive an email link to set a new password
```

**Verify:**
Documentation exists and describes:
- How to create an admin user
- How to log in (development and production)
- How to log out
- How to reset password
- Security considerations

**Done:** Runbook/auth doc updated with admin user management guidance.

### Notes for Production Deployment

When deploying to Vercel/production:

1. **Environment Variables** — Add the same frontend env vars to Vercel project settings:
   - `NEXT_PUBLIC_SUPABASE_URL`
   - `NEXT_PUBLIC_SUPABASE_ANON_KEY`

2. **Admin User** — Create the same admin user in Supabase (it's project-wide, not environment-specific)

3. **Auth Callback URLs** — Verify that Supabase Auth → Settings → Redirect URLs includes your production domain:
   - Example: `https://drift.yourdomain.com/auth/callback`

4. **Testing** — After deployment, test the same login/logout flow on the production URL to confirm auth works.

These will be verified in Phase 5 (Deployment) and Phase 6 (Automation).
