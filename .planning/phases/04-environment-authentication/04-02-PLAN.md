## Plan 04-02: GitHub Secrets & Repository Configuration

**Wave:** 1 (parallel with 04-01, but technically depends on it)
**Autonomous:** Yes
**Duration:** ~15 minutes

### Objective

Store the 4 backend secrets (ANTHROPIC_API_KEY, SUPABASE_URL, SUPABASE_SERVICE_KEY, FIRECRAWL_API_KEY) in GitHub repository secrets. Verify that the agent scheduling workflow (`.github/workflows/agent-schedule.yml`) correctly references them and the YAML is valid.

This plan satisfies **ENV-03** (GitHub Secrets configured for CI/CD).

### Context

**What exists:**
- `.github/workflows/agent-schedule.yml` — workflow file that runs `python backend/agent.py` on schedule
- GitHub repository secrets UI — available at `github.com/{owner}/{repo}/settings/secrets/actions`
- 4 backend env vars from Plan 04-01 (already in local `.env.local`)

**What's needed:**
- Store each of the 4 secrets in GitHub Actions (encrypted at rest, available to workflows)
- Update workflow YAML to pass them to the agent container
- Verify YAML syntax (no errors in GitHub Actions UI)
- Test that the workflow can at least access the secrets (no "undefined" errors)

### Tasks

#### Task 1: Store 4 backend secrets in GitHub Actions

**Files:** (GitHub UI only, no file changes)

**Action:**
Go to GitHub repository settings → Secrets and variables → Actions. For each of the 4 values from `backend/.env.local`, create a new repository secret:

1. **ANTHROPIC_API_KEY** — New secret
   - Name: `ANTHROPIC_API_KEY`
   - Value: (paste from your `backend/.env.local`)

2. **SUPABASE_URL** — New secret
   - Name: `SUPABASE_URL`
   - Value: (paste from your `backend/.env.local`)

3. **SUPABASE_SERVICE_KEY** — New secret
   - Name: `SUPABASE_SERVICE_KEY`
   - Value: (paste from your `backend/.env.local`)

4. **FIRECRAWL_API_KEY** — New secret
   - Name: `FIRECRAWL_API_KEY`
   - Value: (paste from your `backend/.env.local`)

Each secret is encrypted and only accessible within GitHub Actions workflows that have write access. The secrets are masked in logs automatically.

**Verify:**
After adding each secret, GitHub will show a green checkmark. Wait 5 seconds for replication. Refresh the Secrets page and verify all 4 appear in the list (values will be masked as "***").

**Done:** All 4 secrets visible in GitHub Actions → Secrets list (showing masked values).

#### Task 2: Verify workflow references all 4 secrets

**Files:**
- `.github/workflows/agent-schedule.yml` (read, possibly edit)

**Action:**
Open `.github/workflows/agent-schedule.yml` and verify it has an `env:` section or `jobs.<job>.env` that references all 4 secrets using the `${{ secrets.SECRET_NAME }}` syntax.

If the workflow references secrets correctly, it should look like:

```yaml
jobs:
  run-agent:
    runs-on: ubuntu-latest
    env:
      ANTHROPIC_API_KEY: ${{ secrets.ANTHROPIC_API_KEY }}
      SUPABASE_URL: ${{ secrets.SUPABASE_URL }}
      SUPABASE_SERVICE_KEY: ${{ secrets.SUPABASE_SERVICE_KEY }}
      FIRECRAWL_API_KEY: ${{ secrets.FIRECRAWL_API_KEY }}
    steps:
      - uses: actions/checkout@v4
      - name: Run agent
        run: python backend/agent.py
```

If the `env:` section is missing or incomplete, add/update it to include all 4 secrets.

**Verify:**
```bash
grep -A 5 "env:" .github/workflows/agent-schedule.yml | grep -c "secrets\."
# Should output: 4 (one line per secret)
```

**Done:** Workflow YAML explicitly references all 4 secrets via `${{ secrets.SECRET_NAME }}` syntax.

#### Task 3: Validate YAML syntax and GitHub Actions parsing

**Files:**
- `.github/workflows/agent-schedule.yml`

**Action:**
Use the GitHub Actions workflow validator to check for syntax errors. You have two options:

**Option A: Online** (recommended)
1. Go to your GitHub repo
2. Click "Actions" tab
3. Click "agent-schedule" workflow
4. If the workflow shows a green checkmark and "Completed" status, the YAML is valid

If there's a red ✗ and an error message, click on it to see the parsing error. Fix the YAML and commit.

**Option B: Local validation** (if you have the GitHub CLI)
```bash
gh workflow view .github/workflows/agent-schedule.yml --repo {owner}/{repo}
# or
cd .github/workflows
python -c "import yaml; yaml.safe_load(open('agent-schedule.yml'))" && echo "✓ YAML valid" || echo "✗ YAML invalid"
```

**Verify:**
The workflow should show no syntax errors in GitHub's Actions UI. If there are validation errors, they'll be displayed with line numbers.

**Done:** Workflow YAML passes GitHub Actions syntax validation (no red X in Actions UI, or local YAML validator confirms valid syntax).

#### Task 4: Confirm workflow has trigger and schedule defined

**Files:**
- `.github/workflows/agent-schedule.yml` (read)

**Action:**
Verify that the workflow has a `on:` trigger section with `schedule:` defined. The schedule should trigger the agent bi-weekly as per the design.

Expected structure:
```yaml
name: Agent Schedule

on:
  schedule:
    - cron: '0 9 * * 1' # every Monday at 9 AM UTC
    # or another bi-weekly schedule

jobs:
  run-agent:
    runs-on: ubuntu-latest
    ...
```

The exact cron expression may vary, but it should be present and valid.

**Verify:**
```bash
grep -A 3 "on:" .github/workflows/agent-schedule.yml | grep -c "schedule"
# Should output: 1
```

**Done:** Workflow has `on.schedule` trigger defined.

#### Task 5: Document GitHub secrets location and rotation policy

**Files:**
- `docs/RUNBOOK.md` (update "GitHub Secrets" section, if exists)
- or `docs/OPERATIONS.md` (update or create)

**Action:**
Add or update a section in the runbook documenting:

1. **Where GitHub secrets are stored** — `github.com/{owner}/{repo}/settings/secrets/actions`
2. **What secrets are needed** — the 4 backend env vars (ANTHROPIC_API_KEY, SUPABASE_URL, SUPABASE_SERVICE_KEY, FIRECRAWL_API_KEY)
3. **How to rotate/update** — "Go to repo settings → Secrets → edit value → save"
4. **Precedence** — GitHub secrets override local `.env.local` in Actions workflows
5. **Security** — "Secrets are encrypted at rest, masked in logs, never exposed in workflow output"

Example:

```markdown
### GitHub Secrets

The agent workflow reads 4 secrets from GitHub Actions:

- `ANTHROPIC_API_KEY` — Anthropic API key (retrieve from console.anthropic.com)
- `SUPABASE_URL` — Supabase project URL (from Supabase dashboard)
- `SUPABASE_SERVICE_KEY` — Supabase service role key (marked "Secret" in dashboard)
- `FIRECRAWL_API_KEY` — Firecrawl API key (from firecrawl.dev)

**To update a secret:**
1. Go to repo → Settings → Secrets and variables → Actions
2. Click the secret name
3. Paste new value
4. Click "Update secret"

**To add a new secret:**
1. Click "New repository secret"
2. Enter name and value
3. Click "Add secret"

Secrets are available to all workflows on this repository.
```

**Verify:**
Documentation exists and describes the 4 secrets, how to update them, and where to find them.

**Done:** Runbook/operations doc updated with GitHub secrets guidance.

### Security Notes

- Secrets are stored encrypted at rest in GitHub's secure storage
- Each secret is accessible only to workflows in this repository
- Secret values are automatically masked in workflow logs (never appears as plaintext)
- Commits to the workflow file that reference secrets do NOT expose the secret values
- To revoke a secret: delete it from the Secrets UI; it will no longer be available to new workflows
